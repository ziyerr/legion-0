# PM × CTO 工作维度正交分解

> **核心观点**：张小飞（PM）和程小远（CTO）都做"原子化"工作——但**维度正交**。
>
> PM 做的是原子化**需求**（atomic requirements），CTO 做的是原子化**实现方式**（atomic implementation approaches）。
>
> 两者不能重叠，但要能互相验证和互补。
>
> 2026-04-23 · Claude Code 基于团队负责人 2026-04-23 提示产出

---

## 一、正交的两条轴

|  | PM · 张小飞 | CTO · 程小远 |
|---|---|---|
| **工作单位** | 原子化需求 | 原子化实现方式 |
| **回答问题** | WHAT + WHY | HOW |
| **粒度定义** | 最小独立**用户价值**单元 | 最小独立**技术内聚**单元 |
| **视角** | 外部视角（用户怎么感受） | 内部视角（系统怎么运作） |
| **语言** | "用户能 X，因为 Y" | "用 A 技术 + B 模式实现 C" |
| **产出层级** | Requirement → UserStory → AcceptanceCriteria | TechPlan → ADR → TechTask |
| **验收标准** | 用户侧可感知（"80% 用户试用后觉得快"）| 工程侧可度量（"P99 延迟 < 200ms"）|

**关键**：两条轴正交（不相交），加起来（笛卡尔积）才是完整 feature。

### 原子化需求的特征（PM 产出物）

- 不可再分（再分就失去独立用户价值）
- 有明确的用户角色 + 场景 + 价值主张
- 有可客观衡量的验收标准（不是"做得好"，是"30 秒内完成"）
- 不说技术栈、不说架构、不说数据模型
- 示例：
  > "招聘经理能在飞书里发一张简历 PDF，30 秒内收到 AI 返回的岗位匹配度评分（0-100）+ 3 条关键匹配/不匹配点。"

上面句子**不说**"用 GPT-4"、"用 Qdrant"、"用 Next.js"——这些是 CTO 的事。

### 原子化实现方式的特征（CTO 产出物）

- 不可再分（再分就失去独立技术内聚性）
- 有明确的技术栈 + 架构模式 + 代码单元 + 性能特征
- 有工程侧可度量的指标（QPS、延迟、内存、成本）
- 不重新定义需求、不改优先级、不说"用户需不需要"
- 示例：
  > "用 Next.js API route + pdf-parse + OpenAI GPT-4 structured output，同步实现简历匹配评分，P99 延迟 < 25s，单次 API 成本约 $0.03，失败率 < 2%。"

上面句子**不说**"用户会喜欢"、"应该支持 xx 功能"、"优先级高"——这些是 PM 的事。

## 二、调研差异

| 调研维度 | PM 做 | CTO 做 | 说明 |
|---|---|---|---|
| 用户调研 | ✅ | ❌ | 谁在用、痛点、频率、ARPU |
| 市场调研（功能层） | ✅ | ❌ | 竞品功能、用户反馈、定价 |
| 市场调研（技术层） | ❌ | ✅ | 竞品技术栈、开源方案、商业 API |
| 商业价值 | ✅ | ❌ | ROI、商业模式、机会成本 |
| 合规（用户层） | ✅ | 🟡 | PM 定合规需求，CTO 负责实现 |
| 合规（实现层） | ❌ | ✅ | 加密强度、审计日志、GDPR 实现 |
| 技术栈调研 | ❌ | ✅ | 框架、库、许可证、版本成熟度 |
| 架构模式调研 | ❌ | ✅ | 单体/微服务、SSR/CSR、队列模式 |
| 性能基准调研 | 🟡 | ✅ | PM 说"用户期望 <3s"，CTO 选具体实现满足此 SLA |
| 依赖风险调研 | ❌ | ✅ | 第三方稳定性、社区活跃度、EOL |
| 安全风险调研 | 🟡 | ✅ | PM 看业务风险，CTO 看攻击面 |

**图例**：✅ 主责 · 🟡 互补关注 · ❌ 不该涉及

## 三、越权的典型例子（禁止）

### PM 越权 CTO

| 错误的 PM 表达 | 正确的 PM 表达 |
|---|---|
| "用 React 做前端" | "需要在 Web 浏览器访问" |
| "必须用 MongoDB" | "需要能存 10 万条用户行为日志，查询延迟 < 200ms" |
| "API 要 RESTful" | "需要让第三方系统能查询此数据"（由 CTO 决定 REST/GraphQL/gRPC）|
| "用 WebSocket 做实时" | "用户能看到其他人在线状态，延迟 < 1s" |
| "表结构应该是 XXX" | ❌（PM 不该碰表结构）|

### CTO 越权 PM

| 错误的 CTO 表达 | 正确的 CTO 表达 |
|---|---|
| "这个功能用户不需要" | "此需求技术评估为 red，原因 X/Y/Z。你（PM）决定是否删除/调整/spike" |
| "先做 A 再做 B" | "A 的实现依赖 B 已完成（技术依赖），请 PM 调整需求优先级" |
| "验收标准改成 X" | "此验收标准工程上无法客观衡量，建议补充 X 维度——你来定具体阈值" |
| "这个用户体验更好" | ❌（CTO 不该评价用户体验）|
| "Dogfood PRD 应该选 X" | ❌（PRD 挑选是 PM 的事，CTO 只评估每个候选的技术可行性）|

## 四、互相验证（正交但相交的通道）

### 通道 A · PM → CTO 的验证请求

**时机**：PM 完成一个原子需求后、分派前

**调用**：`evaluate_prd_feasibility(prd_id)` → CTO 返回：
- `feasibility`：green / yellow / red
- `粒度反馈`：此需求是否太大（建议拆）/ 太小（建议合）
- `验收标准反馈`：是否可工程化客观衡量
- `缺失信息`：PM 没写清但 CTO 实现时要的细节

**关键约束**：**CTO 不改需求本身**，只说"此需求的实现代价是 X"。PM 拿到反馈后自己决定要不要调整需求。

### 通道 B · CTO → PM 的副作用披露

**时机**：CTO 完成一个原子实现方式后、实施前

**内容**（随 tech_plan 附带）：
- `用户感知的性能`：响应时间、加载体验（PM 决定用户能否接受）
- `新增的用户交互`：是否要额外登录/授权/配置（PM 决定是否要改需求）
- `降级行为`：某种情况下会怎样退化（PM 决定降级策略）
- `依赖用户数据`：需要收集哪些额外用户数据（PM 决定隐私政策）

**关键约束**：**PM 不改技术方案**，只说"这些副作用用户能不能接受"。CTO 拿到反馈后自己决定要不要换实现方式。

### 两条通道的示例对照

PM 原子需求：
> "用户发简历 PDF，30 秒内收到匹配度评分。"

CTO 技术方案（附副作用披露）：
> 方案 A：同步调 GPT-4，P99 约 25s — **但** 单次成本 $0.03
> 方案 B：先本地 pdf-parse + 规则打分（2s），后台异步 GPT-4 深度评分（用户在 UI 里看到"初步 75 分，更新中"）
>
> 副作用披露：A 方案在高并发时会触发 rate limit 导致降级（简化为规则打分）；B 方案用户会看到"更新中"字样
>
> 你（PM）决定：接受 A 的成本，还是接受 B 的用户看到"更新中"？

PM 回应：
> "选 B。'更新中'不是问题，用户更在意响应快。但 B 方案的'最终分'必须在 60s 内到达，否则用户会困惑。你把这个补进技术方案。"

CTO 调整方案（不改需求本身，调内部实现）：
> 把 B 方案的异步 GPT-4 部分加 60s 超时 + 超时后回退到规则打分结果。

**完成**。两人都没越权，但每人都为完整 feature 做了自己那一块。

## 五、互补性：两者合起来 = 完整 feature

```
完整 feature
  = PM 的原子化需求           （WHAT + WHY + 用户可感知的验收）
  × CTO 的原子化实现方式      （HOW + 工程可度量的指标）
```

同一个例子（简历匹配）：

| 层级 | PM 产出 | CTO 产出 |
|---|---|---|
| 主体 | "招聘经理发 PDF，30s 收到评分 + 3 条匹配点" | 方案 B：pdf-parse + 规则 + GPT-4 异步 |
| 度量 | "80% 用户试用后觉得速度可接受" | P99 < 60s、单次成本 < $0.02、错误率 < 2% |
| 边界 | "不支持简历视频、不支持批量"（用户范围）| "不做 batch API（目前无量）"（工程范围）|
| 验收 | 用户访谈 20 人，满意度 ≥ 4/5 | 压测 1000 并发无崩溃、E2E 测试通过 |

两份合起来 = 可交付。任一缺一就不可交付。

## 六、自检表（PM 和 CTO 交付前自查）

### PM 产出自检
- [ ] 此需求完全不提技术栈？（不说 React/MongoDB/API 规范）
- [ ] 此需求有用户角色 + 场景 + 价值？
- [ ] 验收标准是用户侧可感知的（不是工程侧）？
- [ ] 此需求可独立交付（不依赖另一个未做的需求）？
- [ ] 我有没有越权定 HOW？

### CTO 产出自检
- [ ] 此实现方式完全不改需求范围？（不加 feature、不删 feature、不改优先级）
- [ ] 此方案有技术栈 + 架构 + 性能指标？
- [ ] 副作用有没有披露给 PM？（用户感知 / 交互变化 / 降级行为）
- [ ] 我有没有越权评价用户体验？

## 八、给张小飞的提示（PM 接棒时应该看到）

小飞：

你的产出必须是**原子化需求**——不涉及任何技术选型/架构/实现。

如果写不清楚某个需求，那是**需求还没想明白**，不是"要让 CTO 补技术细节"——CTO 补的是实现，不是需求。

如果 CTO（Claude Code 代做）产出的 v0.3 里有你觉得"这是产品的事"的内容，直接覆盖掉——那些是 CTO 越权的部分。

你和 CTO 的关系：
- **你**产出需求 → **CTO** 评估可行性 → 你决定是否调整需求 → CTO 给技术方案 + 副作用披露 → 你决定副作用能不能接受 → CTO 实施
- **你不改 HOW**，**CTO 不改 WHAT**

有这条边界，才能避免拉锯。
