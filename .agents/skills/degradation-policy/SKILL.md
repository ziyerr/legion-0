# 降级原则（第一性原理）

## 核心规则

写 try/except、fallback、skip 之前**必须先判断**：

1. **这个环节的目的是什么？**（不是"它做什么"，而是"为什么需要它"）
2. **跳过它会影响用户的核心目标吗？**
3. **影响 → 不可降级**：尝试修复 → 重试 → 溯源补全 → 问询用户
4. **不影响 → 可降级**：记录 warning，继续执行

**禁止盲目 try/except + skip。每个降级都必须有理由。**

## 示例

| 场景 | 判断 | 处置 |
|------|------|------|
| 素材缺失 | 影响一致性 | 不可跳过 → 溯源生成或问用户 |
| 封面图失败 | 不影响视频 | 可跳过 → warning |
| 角色三视图缺失 | 影响角色一致性 | 不可跳过 → 必须先生成 |
| 字幕生成失败 | 影响交付质量 | 不可跳过 → 修复 |
| 缩略图生成失败 | 不影响核心 | 可跳过 → warning |

## Gotchas

1. **"为了不阻塞流程"不是降级理由** — 阻塞也比交付残缺好
2. **降级决策要记录** — 在代码注释中说明为什么可以降级
