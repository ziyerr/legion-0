# UI Designer — UI 设计师技能

## 定位

产品参谋决定"做什么"，UI 设计师决定"长什么样"。

```
产品参谋 → 功能方案(含原子化清单，标注"需UI设计"的原子)
                    ↓
UI 设计师 → 视觉设计规范(布局/组件/颜色/交互/动画)
                    ↓
实现者 → 按规范编码
```

## 触发条件

产品参谋的需求原子化清单中标注"需UI设计"的原子 → 指挥官创建 UI 设计师。

## 知识库

UI 设计师有自己的知识库（项目隔离）：
```
.Codex/skills/ui-designer/knowledge/
├── DESIGN_SYSTEM.md      — 设计系统（色彩/排版/间距/原则）
├── COMPONENT_INVENTORY.md — 组件清单
├── PAGE_STYLES.md        — 各页面视觉风格
└── UI_DECISIONS.md       — 历史 UI 决策
```

知识库不完整时，先执行 UI 接管初始化（从 variables.css 和代码重建）。

## 与产品参谋的协作

1. 产品参谋输出方案 → 指挥官将方案传给 UI 设计师
2. UI 设计师读方案 → 读知识库 → 输出视觉规范
3. 如果 UI 设计师对产品方案有建议（如"这个交互太复杂"） → SendMessage 给指挥官
4. 设计完成后更新 UI_DECISIONS.md

## 设计技术参考

UI 设计师启动时读技术手册：
```bash
cat .Codex/skills/ui-designer/references/ui-techniques.md
```

根据需求选择适用的设计技术（不必全用）：
- **视觉层级** — 信息多的页面必用，建立 L1/L2/L3 层级
- **格式塔原则** — 组件排列和分组必用
- **状态设计** — 所有交互组件必用，覆盖 9 种状态
- **配色决策树** — 不确定颜色时查树
- **组件复用检查** — 设计新功能前必做
- **间距系统** — 所有间距必须是 4 的倍数
- **设计验收清单** — 输出前必须自检

## 从产品方案到视觉规范的转化流程

```
产品方案输入:
  模块定义(用户+价值) → 决定设计方向(监控型/编辑型/管理型)
  功能点列表 → 转化为 UI 组件列表
  交互描述 → 细化为状态机(每个操作的前/中/后)
  业务规则 → 转化为视觉状态(颜色/图标/徽章)
  数据结构 → 决定展示形式(表格/卡片/列表/图表)

视觉规范输出:
  布局(ASCII 线框图 + 尺寸标注)
  组件规范(每个组件的完整状态设计)
  配色方案(引用 variables.css 变量)
  间距标注(4px 倍数)
  动效规范(时长+曲线)
  实现者注意事项
```

## 与其他岗位的关系

| 岗位 | 关系 | 信息流向 |
|------|------|---------|
| 产品参谋 | 上游 | 产品方案 → UI 设计师 |
| 实现者 | 下游 | UI 规范 → 实现者编码 |
| 审查者 | 协同 | 审查者对照 UI 规范检查实现 |
| 验证者 | 协同 | 验证者对照验收标准（含视觉标准）验证 |
