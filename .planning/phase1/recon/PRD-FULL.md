## 一、一句话描述

为云智 OPC 团队解决 PM 越权调度军团、无人做技术决策和代码审查问题的 AI 技术总监（CTO Agent）。

## 二、背景与动机

### 为什么做

当前组织架构存在结构性缺陷：PM Agent（张小飞）通过 dispatch_to_legion 直接给开发军团派活，跳过技术评审环节。骏飞（team lead）每天被迫花 1-2 小时人工介入 3-5 次技术决策（技术栈选型、跨军团协调、PR 审查），不可持续。各军团独立工作无人做整体架构把关，返工成本约占 30-50%（行业数据）。

### 为什么现在做

2026 年 Claude Opus 4.6/4.7 的长上下文 + 高推理 + tool use 能力让 CTO 级别 AI Agent 首次可行（Anthropic 2026 Agentic Coding Trends Report）。军团基础设施刚成熟（mailbox/outbox 通讯协议就绪），ProdMind dev.db 共享数据层已有项目/PRD/任务表可复用。

### 不做会怎样

- 骏飞继续当人肉 CTO，时间被锁死在日常技术决策上
- PM 继续越权派活，军团收到的任务质量无保障
- 多项目并行时跨军团冲突和返工越来越频繁
- OPC 闭环缺关键角色，无法对外输出完整方案

## 四、产品目标 OKR

**O**：建立 PM → CTO → 军团的技术管理闭环，释放 team lead 的技术决策负担

| KR | 衡量方式 | 基线 | 目标 |
|---|---|---|---|
| KR1：骏飞技术介入频率下降 | 每周技术决策次数 | 15-25 次/周 | ≤5 次/周 |
| KR2：代码审查覆盖率 | 军团 PR 经 CTO 审查比例 | 0% | 100% |
| KR3：返工率下降 | 军团任务返工比例 | 估 30-50% | ≤15% |
| KR4：技术方案产出速度 | PRD → 技术方案时间 | 骏飞 1-2 小时 | ≤5 分钟 |

## 五、功能规格

### P0 — Phase 1 必须（闭环最小集）

#### 能力 0：项目启动自动化（kickoff_project）

**描述**：PM 发起新项目后，程小远自动完成 8 步：创建项目目录 → git init → ProdMind 项目条目 → ADR-0001 → 拉军团 → 建通讯 → 派首批任务 → 飞书群通知。

**用户价值**：team lead 不再需要手动搭项目骨架，从"PM 说要做"到"军团就位"全自动。

**触发**：PM 调用 create_project 或飞书 @程小远 说"启动项目 X"

**输出**：结构化 JSON（project_id / git_path / legion_commander_id / adr_id / initial_tasks）+ 飞书卡片通知

**飞书卡片 UX**：

```
┌──────────────────────────────────────┐
│ 🚀 项目「AICS」启动                  │
├──────────────────────────────────────┤
│ 📍 Path: ~/Documents/AICS           │
│ 🏛️ Legion: L1-AICS-后端 (就位)      │
│ 📝 ADR-0001 已记录                   │
│ ⏳ 等 PM 发 PRD 启动首批任务         │
├──────────────────────────────────────┤
│ [查看 ADR]  [加入军团群]  [暂停项目]  │
└──────────────────────────────────────┘
```

**失败处理**：4 级错误分类（技术→自主重试 / 权限→升级骏飞 / 意图→给选项 / 未知→保守升级）

#### 能力 1：技术方案设计（design_tech_plan）

**描述**：读 PRD → 评估可行性 → 选技术栈 → 设计架构 → 识别风险 → 输出飞书文档。

**用户价值**：PM 发 PRD 后 5 分钟内拿到技术方案，不再等骏飞有空。

**输入**：prd_id 或 prd_markdown + 可选 focus/constraints

**输出**：feasibility（green/yellow/red）+ 技术栈选型 + 时间估计（乐观/可能/悲观）+ 风险清单 + 缺失信息 + 飞书文档 URL

**关键规则**：

- red verdict 必须告诉 PM 改什么才能变绿（不能只说不可行）
- missing_info 反向推回 PM 澄清，过不了就不能进任务分派
- 每个技术选型决策自动写入 ADR 表

---

#### 能力 2：任务拆分（breakdown_tasks）

**描述**：技术方案 → 结构化任务列表 + 依赖 DAG + 验收标准

**用户价值**：军团指挥官收到的任务读完就能动手，不需要额外澄清。

**关键规则**：

- 单任务不超过 XL（≥3 天必须再拆）
- 依赖关系强制 DAG（不允许环）
- 每个任务的验收标准必须 Given/When/Then
- 按 EngineerProfile（未来）推荐军团

---

#### 能力 3：军团调度（dispatch_to_legion_balanced）

**描述**：根据任务技术栈 × 军团能力 × 军团当前负载，智能分配任务。

**用户价值**：军团不会过载，有依赖的任务自动排序，PM 不再越权派活。

**关键规则**：

- 单军团同时 ≤2 个任务
- 有依赖关系的任务延迟派单
- 派单时附完整上下文（PRD 摘要 + 技术方案 + 验收标准）
- CTO 拥有调度决策权，军团必须接（可 appeal 但不可直接拒）

---

#### 能力 4：代码审查（review_code）

**描述**：10 项审查清单，每项 PASS / BLOCKING / NON-BLOCKING，硬 gate 阻塞 merge。

**用户价值**：代码质量有底线保障，骏飞不需要逐行看 PR。

**10 项清单**：

| # | 维度 | 检查点 |
|---|---|---|
| 1 | 架构一致 | 是否符合技术方案？ |
| 2 | 可读性 | 命名规范、代码清晰？ |
| 3 | 安全 | 有无安全漏洞？ |
| 4 | 测试 | 关键路径有覆盖？ |
| 5 | 错误处理 | 边界情况处理？ |
| 6 | 复杂度 | 有无不必要复杂性？ |
| 7 | 依赖 | 第三方依赖合理？ |
| 8 | 性能 | 性能可接受？ |
| 9 | 跨军团冲突 | 与其他军团代码冲突？ |
| 10 | PRD 一致 | 满足验收标准？ |

**BLOCKING 硬 gate 规则（骏飞 2026-04-23 拍板）**：

- BLOCKING = 真阻塞 merge，军团必须停 + 修 + 重 PR
- BLOCKING 必须附明确修复要求（不是"这里不好"而是"把 X 改成 Y 因为 Z"）
- 军团忽略 BLOCKING = 执行纪律违规，自动升级骏飞

**Appeal 通道**：

- 军团觉得 BLOCKING 不合理 → 提 appeal
- 程小远收到 appeal → 要么收回 BLOCKING 并说明理由，要么维持并升级骏飞仲裁
- Appeal 飞书卡片：

```
┌──────────────────────────────────────┐
│ ⚠️ BLOCKING Appeal                   │
├──────────────────────────────────────┤
│ PR: #42 - 用户认证模块               │
│ BLOCKING: 缺少 CSRF 防护             │
│ 军团理由: 内部工具不需要 CSRF         │
├──────────────────────────────────────┤
│ [维持 BLOCKING]  [收回]  [升级骏飞]   │
└──────────────────────────────────────┘
```

**评论密度限制（防过度审查）**：

- 单 PR 最多 5 个评论（超出按 severity 排序只留 top 5）
- 单文件最多 2 个 BLOCKING（超出建议整体 refactor）

---

#### 能力 5：进度汇报（daily_brief + escalate）

**描述**：每日 18:00 技术进度摘要 + BLOCKING 即时推送 + 军团 >24h 无进展自动催促。

**用户价值**：骏飞每天 30 秒看一眼群消息就掌握全部技术进度，不需要主动问。
