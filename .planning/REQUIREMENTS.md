- R22: Protect all mixed registry mutations with `flock`-backed transactions that re-read under lock, validate monotonic status transitions, and commit dependency/lifecycle side effects atomically.
- R23: Protect inbox and event-log writes with `flock`; `mixed msg` and `mixed broadcast` must persist the inbox record before tmux notification, and tmux failure must not erase durable delivery.
- R24: Harden readiness collection with a fresh readiness order boundary, direct-L2 registry validation, structured order/nonce fields, and sender identity that cannot be forged through arbitrary CLI `--sender` strings or free-text inbox rows.
- R25: Split read-only status from mutating reconcile/repair, and classify tmux liveness errors before any registry mutation.
- R26: Gate the communication upgrade on deterministic registry race tests, inbox/event durability tests, tmux delivery-order tests, readiness trust-boundary tests, hierarchy delivery tests, and explicit release evidence in `STATE.md`.
- R27: Lock the no-omission legacy parity contract. Every Claude-only Legion mechanism in commit `137d861` must be retained as-is, upgraded in mixed, replaced by a stronger mixed mechanism, or recorded as a blocker with a named owner and verification path. Documenting a mechanism in `.planning/` does not satisfy the parity requirement; runtime evidence must back every "retained" or "upgraded" claim before release.
- R28: All mixed registry mutations (`_upsert_task`, `_upsert_commander`, `_set_task_status`, commander refresh, dependency block/unblock, lifecycle/disband, mark, complete, reconcile/repair, readiness-order metadata) must run inside one shared registry transaction helper using `fcntl.flock` with reread-under-lock, monotonic-state validation, and atomic temp+replace while still holding the lock. Lock placeholder files alone do not satisfy this requirement; runtime code must invoke the helper.
- R29: All inbox JSONL appends and read-cursor updates must run under a per-inbox `flock`. Every inbox record must carry a stable message id, sender id, recipient id, timestamp, type, content, and optional `reply_to`, `order_id`, and `requires_ack`. Free-text rows without these fields must not satisfy any contract that requires structured payload.
- R30: All `events.jsonl` appends must run under an event-log `flock`. Every event must carry a stable event id, schema version, type, timestamp, and a correlation id linking to the registry transaction, inbox message, dependency edge, or readiness order it represents. Empty-payload status events do not satisfy the audit-trail requirement.
- R31: `mixed msg` and `mixed broadcast` must persist the durable inbox record before any tmux notification attempt. If inbox persistence fails, no tmux notice may be sent and no success event may be appended. If tmux notification fails, the inbox record remains authoritative and the event records `delivered_tmux=false`. Tmux delivery for commander messages must be non-invasive notification, never prompt injection. Broadcast must take recipient snapshots from a locked registry view and persist one idempotent inbox record per target.
- R32: Readiness collection must scope to a current readiness order recorded in the locked registry. Each order carries `order_id`, `issued_at`, parent id, expected direct L2 ids, and a per-order nonce. A readiness reply counts only if it is newer than the order freshness boundary, echoes the current `order_id` and nonce, comes from a sender registered as a direct L2 child of the parent in an active allowed status with a current session/run binding, and is delivered through structured fields rather than free-text `READY:init-complete`. Sender identity must be derived from the commander's execution context, not from an arbitrary `--sender` CLI string. Historical inbox rows from before the current order must not count as readiness.
- R33: `mixed status` must be read-only and must not invoke `reconcile_state()` or any registry mutation. Reconciliation/repair must be an explicit command, route every mutation through the registry transaction helper, and emit locked events. Tmux probe outcomes must classify each session as live, missing, or inaccessible (permission/socket failure) before any registry mutation; "Operation not permitted" must not be collapsed into "missing" and must never mark a live commander failed.
- R34: Worker completion must be structured and provider-agnostic. A Claude or Codex process exit 0 without a parseable success result must mark the task `blocked` or `failed`, not `completed`. Tasks marked completed without a result file containing an explicit success marker must not satisfy any downstream dependency. Replacement of a previously-failed task must explicitly repair the dependency edge through a recorded repair event before downstream blocked tasks may launch.
- R35: Legacy shell, patrol, gate, and retrospector surfaces must not run as a parallel split-brain over their own state files. `legion.sh mixed` must either bridge legacy `locks`, `board`, `sitrep`, `watch`, `mailbox`, `patrol`, `gate`, and `retrospector` reads/writes into the mixed registry/events/inboxes contract, or expose mixed-native equivalents that the release gate consumes. Missing bridges are blockers, not silent skips, and the release gate must enumerate them.
- R36: File-scope ownership must be enforced, not just recorded. Implementation, product, and UI tasks must declare a normalized scope; the registry transaction must detect overlap with any other active implementation/product/UI scope and refuse to mark the conflicting task `running`, recording a scope-conflict block until the prior holder releases. Review, verify, and audit branches must be able to inspect scope-lock records.
- R37: The no-omission release gate is fail-closed. The final release matrix must enumerate every row from the legacy mechanism coverage table and assign one of: retained-as-is (with runtime evidence), upgraded-in-mixed (with runtime evidence), replaced-by-stronger-mixed (with runtime evidence and named replacement), or blocked-with-repair-task (with owner, verification path, and target task id).
