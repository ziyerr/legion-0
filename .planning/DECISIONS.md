# Legion System Repair Decisions

- [LOCKED] Preserve existing local/untracked user changes unless directly required for this repair.
- [LOCKED] Use file-isolated worker agents for implementation, then central verification.
- [LOCKED] Prefer small, deterministic fixes over broad architecture rewrites in this pass.
- [LOCKED] No-omission legacy parity is a release-blocking contract. Reference: `.planning/communication-upgrade/04-LEGACY-PARITY-MATRIX.md`. Every legacy mechanism in commit `137d861` (tmux visibility, L1/L2 hierarchy, branch roles, file scopes, registry/locks/taskboard, inbox/outbox + broadcast + SendMessage discipline, dependencies, approvals/gates, patrol, inspector, review, verify, audit, retrospector, observations, metrics, protocol evolution, tactics, generated skills, heartbeat, GC, war-room/status UX, install/hooks, docs) must end up in one of four states: retained-as-is, upgraded-in-mixed, replaced-by-stronger-mixed, or blocked-with-repair-task. The "blocked" state requires a named owner and a verification path. Silent omission is forbidden.
- [LOCKED] Docs-only runtime support is not implementation. Writing a contract into `.planning/communication-upgrade/00-RUNTIME-CONTRACT.md` does not constitute a "completed" entry in `STATE.md`. The mechanism must be enforced by `scripts/legion_core.py` / `scripts/legion.sh` / hooks at runtime and proven by a verification command before being marked complete. Reverse claims must be relisted as residual risks or blockers.
- [LOCKED] Registry, inbox, and event-log writes must use `fcntl.flock` through the restored runtime helpers. The registry helper acquires the lock, re-reads the latest state, validates monotonic transitions, mutates in-memory, writes via temp+atomic replace while still holding the lock, then releases; inbox/events append through the append lock. Placeholder `.lock` files alone never satisfy this decision, and any mutation path outside these helpers reopens the runtime blocker.
- [LOCKED] Inbox-before-tmux is the delivery order. `mixed msg` and `mixed broadcast` must persist the durable inbox record before any tmux notification attempt; tmux failure must record `delivered_tmux=false` without erasing the inbox. Tmux for commander notifications uses non-invasive `display-message` (or equivalent), never prompt injection.
- [LOCKED] Readiness trust requires order-id, nonce, freshness, direct-L2 registry validation, parent match, active session/run binding, and non-forgeable sender derived from execution context. Free-text `READY:init-complete` and arbitrary `--sender` strings do not satisfy this decision. Historical inbox rows from before the current order must not count.
- [LOCKED] `mixed status` is read-only. Reconcile/repair is a separate explicit command. Tmux probe outcomes are classified live / missing / inaccessible before any state mutation. "Operation not permitted" must never collapse into "missing" or mark live commanders failed.
- [LOCKED] Worker completion is structured for all providers. Claude exit 0 without a parseable success result is `blocked`/`failed`, not `completed`. Tasks completed without a structured success marker must not satisfy downstream dependencies, and replacement completions must repair dependency edges through a recorded repair event.
- [LOCKED] Legacy shell, patrol, gate, mailbox, board, sitrep, watch, and retrospector surfaces must be bridged into mixed registry/events/inboxes or replaced by mixed-native equivalents the release gate consumes. Parallel split-brain operation over legacy state files is forbidden; missing bridges are blockers and the release gate must enumerate them.
- [LOCKED] File-scope ownership is enforced inside the registry transaction. Implementation/product/UI scopes are normalized, conflicting active scopes block the new task with a scope-conflict record, and review/verify/audit branches inspect those records. Recording scope without enforcing it is not parity.
- [LOCKED] The no-omission release gate is fail-closed. Any blocked row, any unverified upgrade claim, and any docs-only "implemented" claim forces the final verdict to FAIL or WATCH. PASS requires runtime evidence for every row and zero remaining blockers, or an exact STATE.md entry naming the residual blocker, owner, and repair task.
- [LOCKED] L1 no-delivery is now complexity-scoped. S-level work may stay with the receiving L1; M+ deliverables must pass through a direct L2 branch commander or L2-supervised worker via `--corps`.
- [LOCKED] Provider strengths drive defaults: Claude remains the default for implementation/product/UI and long-context delivery; Codex remains the default for explore/review/verify/audit/security. If Codex is explicitly assigned a write-capable delivery task, the plan must set `sandbox=workspace-write`; Codex quality gates stay read-only by default.
- [LOCKED] Dry-run is non-mutating. A dry-run may render planned commanders, tmux scripts, routing decisions, and scope conflicts, but must not write registry, events, inbox, run directories, or refresh/reuse state.
- [LOCKED] Repair is an explicit state transition. Failed/blocked tasks remain terminal; replacement tasks are validated before mutation, downstream dependency repair emits a `task_repair` event, and invalid repair attempts leave registry/events byte-for-byte unchanged.
- [LOCKED] Release evidence must include durable artifacts, not prose claims: locked registry/inbox/events rows, strict worker result files, patrol status/gate evidence, retrospective/after-action inputs, and independent review/verify/audit outputs.
- [LOCKED] Legacy daemon, inspector, hook, patrol, and observations evidence remains part of the release contract. Mixed-native replacements must consume equivalent or stronger evidence; otherwise the missing bridge is recorded as WATCH/FAIL rather than omitted.
- [LOCKED] Dual-L1 is the normal mixed command topology. Claude L1 owns Claude branches and Codex L1 owns Codex branches; Legion Core is the L0 coordination layer. Cross-provider work is coordinated by durable orders/results and synthesis gates, not by a single L1 sending control messages as another provider context.
- [LOCKED] Broadcast fan-out is live-roster-only for control-plane use. Registry `commanding` is not enough; tmux liveness and parent/project binding must agree before a commander is treated as a broadcast recipient or visible child.
- [LOCKED] AICTO is the external Hermes CTO project (`/Users/feijun/Documents/AICTO`), not a local Legion Core L0 commander. Legion Core provides the Claude/Codex army runtime; AICTO commands it through Hermes plugin tools, durable inbox/events/results, and evidence-bearing directives.
- [LOCKED] Codex team management uses the same tmux-backed Legion Core path as Claude: Codex L1 and L2 commanders get visible tmux sessions, Codex workers get visible mixed tmux windows, and no Codex subteam may be launched as an invisible background agent outside `mixed campaign`.
- [LOCKED] Default public startup is stable dual-L1 only. `legion h` / `legion host` starts Claude L1 and Codex L1, sends a 1-second-delayed durable peer-sync between them, and does not create base L2 commanders unless an explicit legacy/base-L2 path requests it.
- [LOCKED] M+ is the expansion boundary. S-level AICTO/L1 directives stay with L1; M/L/XL work expands through `mixed campaign --corps` into visible provider-owned L2/team tmux windows.
- [LOCKED] L2 startup is lightweight and task-scoped. L1/Legion Core must activate L2 with target/scope/dependency context; L2 loads only relevant project experience, tools, skills, and pre-research needed to complete that task instead of repeating full L1 initialization.
- [LOCKED] AICTO reporting is durable outbox, not local L0 resurrection. L1 online/resume, task terminal states, and manual problem reports append to `aicto-reports.jsonl` and events; same-project L1 peer notices use mixed inbox messages.
- [LOCKED] Claude/Codex L1 visibility is separate-session first. Host startup may launch both L1 commanders and peer-sync them, but it must attach only the requested provider L1 in the current terminal and leave the peer L1 as a separately attachable tmux session. Combined tmux views are explicit operator views, not startup defaults.
