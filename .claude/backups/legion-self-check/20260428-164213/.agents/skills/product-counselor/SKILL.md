# Product Counselor — 产品参谋技能

## Overview

**铁律第5条：所有功能开发必须经产品参谋设计并按其完善后的需求开发。**

覆盖范围：新功能 / 改功能 / 新UX / 改交互 / 改架构逻辑 / 改字段 / 设计缺陷修复。仅纯代码质量修复跳过。

产品参谋在团队中的定位：**不是可选的补充，而是功能开发的必经门。**

## 产品接管初始化

产品参谋首次被召唤或知识库不完整时，必须先执行接管初始化：

```
检查 .Codex/skills/product-counselor/knowledge/ 下的文件:
  PRODUCT_BRIEF.md — 不存在或为空 → 从代码和文档重建产品定位
  FEATURE_MAP.md   — 不存在或为空 → 遍历 gui/src/pages/ + v2/skills/ 重建功能清单
  DECISIONS.md     — 不存在或为空 → 从 MEMORY.md 和战法库提取历史决策
  USER_FEEDBACK.md — 不存在或为空 → 创建空模板
  ROADMAP.md       — 不存在或为空 → 从 project_pending_tasks.md 提取规划
```

初始化完成后才能处理具体需求。这确保产品参谋始终基于完整的产品认知工作，而非从零猜测。

```
需求 → [recon 技术侦察] → [产品参谋 产品设计+需求原子化] → [spec-driven 规格文档] → [agent-team 按原子并行实现]
                              ↑ 本技能
```

产品参谋的标准输出必须包含**需求原子化清单**——将设计方案拆为最小独立可交付单元，标注依赖关系和并行编排建议。指挥官据此分配 agent-team 并行实现。

## 触发条件

当需求满足以下**任一**条件时，指挥官应邀请产品参谋加入：

1. **技术简单但产品模糊** — 知道怎么做，但不确定做成什么样
   - 例："加个导出功能" → 导出什么格式？导出哪些数据？导出到哪里？
2. **涉及用户交互设计** — 需要定义操作流程、反馈方式、异常处理
   - 例："支持批量操作" → 选择方式？进度反馈？部分失败怎么办？
3. **涉及业务规则** — 需要定义条件、状态流转、边界情况
   - 例："自动重试失败任务" → 重试几次？间隔多久？什么条件不重试？
4. **需求与现有功能有交集** — 可能影响已有功能的行为
   - 例："新增视频预览" → 和现有播放器什么关系？能否复用？
5. **用户明确要求** — "帮我想想这个功能怎么设计"

## 不触发的情况

- 纯 bug 修复（行为已明确，修就完了）
- 纯技术重构（不改变产品行为）
- 需求已有完整 PRD 或设计稿
- 用户说"不用设计，直接做"

## 与 brainstorming 的边界（二者不是二选一，而是前后衔接）

```
brainstorming（发散）→ product-counselor（收敛）→ spec-driven（落地）

brainstorming:  用户有模糊想法 → 探索意图 → 提出 2-3 种方案 → 用户选定方向
                ↓ 方向确定后
product-counselor: 选定方案 → 结合产品现状细化 → 业务规则/交互/数据/验收标准
                ↓ 设计确定后
spec-driven:    写入 .planning/ → 团队执行
```

**简单判断：**
- 用户说"我想做个 XX" 但没说清楚要什么 → **brainstorming**（先发散探索）
- 用户说"做 XX"，方向明确但细节需要设计 → **product-counselor**（直接收敛）
- 两者都需要 → brainstorming 先跑，产出方向后交给 product-counselor 细化

**禁止：** 两个同时启动做同一件事。一个发散一个收敛，不要重复劳动。

## 邀请方式

产品参谋通过 `subagent_type: "product-counselor"` 创建，作为团队成员之一。

### 独立邀请（最常见）

需求到手 → 指挥官判断需要产品设计 → 邀请产品参谋 → 收到设计方案 → 进入实现阶段。

```
Agent(
  subagent_type: "product-counselor",
  name: "pm-scout",
  model: "opus",
  prompt: """
    首先执行：/effort max

    你是产品参谋，负责为以下需求输出产品设计方案。

    【需求】
    {指挥官填写需求描述}

    【背景】
    {指挥官填写相关上下文，如用户反馈、已有功能、已知约束}

    读 .Codex/agents/product-counselor.md 了解你的职责和输出格式。
    完成后将产品设计方案通过 SendMessage 发给指挥官。
  """
)
```

### 与技术参谋并行（L/XL 级需求同时需要产品设计和技术侦察）

```
指挥官同时创建：
├── 参谋-技术（subagent_type: "explore"）→ 怎么做
├── 参谋-风险（subagent_type: "explore"）→ 有什么坑
└── 参谋-产品（subagent_type: "product-counselor"）→ 做成什么样
```

三路报告汇总后，指挥官综合决策：产品参谋的设计方案 + 技术参谋的实现路径 + 风险参谋的避坑指南 → 写入 `.planning/`。

## 指挥官综合流程

```
1. 接收需求
2. 三步深挖（苏格拉底 + 第一性原理 + 奥卡姆剃刀）
2.5 需求完整性检查 → 若需求模糊（<15字/缺动词/含"优化/改进/一下"/缺对象）→ agent 执行
    Step 0.5 interview 一轮澄清（详见 ~/.Codex/agents/product-counselor.md Step 0.5）
3. 判断：技术简单但产品需要设计？
   ├── 是 → 邀请产品参谋（可与技术参谋并行）
   └── 否 → 走标准 recon 流程
4. 收到产品设计方案
5. 审阅方案，标注 ⚠️ 的地方需要问用户确认
6. 方案确认后 → 写入 .planning/REQUIREMENTS.md
7. 进入 agent-team 实现阶段
```

## 产品参谋的知识来源

产品参谋启动后会自动读取以下信息建立产品认知：

| 来源 | 内容 | 路径 |
|------|------|------|
| 项目记忆 | 产品背景、历史决策、用户偏好 | `~/.Codex/projects/*/memory/` |
| 全局战法 | 已踩过的产品/业务坑 | `~/.Codex/memory/tactics/` |
| 前端页面 | 当前 UI 结构和交互 | `gui/src/pages/`, `gui/src/components/` |
| 后端命令 | 当前 API 能力 | `gui/src-tauri/src/commands/` |
| V2 流水线 | 当前 AI 流水线能力 | `v2/skills/`, `v2/agents/` |
| 项目数据 | 实际运行数据结构 | `projects/` |

## 与其他角色的协作边界

| 角色 | 关注点 | 产品参谋的交集 |
|------|--------|---------------|
| 参谋-技术 | 怎么实现 | 产品参谋定义"做什么"，技术参谋定义"怎么做" |
| 参谋-风险 | 有什么坑 | 产品参谋关注产品层面的风险（用户困惑、功能冲突） |
| 架构师(Plan) | 技术设计 | 产品参谋的方案是架构师的输入 |
| 审查者(Review) | 代码质量 | 审查者可引用产品参谋的验收标准来判断实现是否正确 |

## 持续开发中的产品参与

产品参谋不仅在侦察阶段出现。在持续开发中：

**实现阶段回检**: 实现者完成后，如果审查者发现实现与产品方案偏离，指挥官应重新召唤产品参谋确认。

**验证阶段回检**: 验证者做合规审计时，对照产品方案的验收标准逐条检查。产品方案中的验收标准是验证者的检查清单之一。

**迭代反馈**: 用户对功能提出修改意见时，先经产品参谋评估影响范围和方案调整，再进入技术实现。不直接改代码。

**迭代时的知识库更新（产品参谋必做）：**
1. 更新 `.planning/PRODUCT_DESIGN.md`（修改方案）
2. 更新 `knowledge/DECISIONS.md`（记录为什么改、否决了什么）
3. 更新 `knowledge/USER_FEEDBACK.md`（记录用户反馈）
4. 更新 `knowledge/FEATURE_MAP.md`（如果功能状态变化）
5. 通知 UI 设计师（如果 UI 需跟着改）→ 指挥官重新召唤 UI 设计师更新 `UI_SPEC.md`

**UI 迭代联动：** 产品方案变更涉及前端时，UI 设计师必须同步更新：
1. `.planning/UI_SPEC.md`（更新规范）
2. `knowledge/UI_DECISIONS.md`（记录为什么改）
3. `knowledge/COMPONENT_INVENTORY.md`（如果组件变化）
4. `knowledge/PAGE_STYLES.md`（如果页面风格变化）

流程闭环:
```
用户需求 → 产品参谋(方案) → UI设计师(规范) → 实现 → 审查 → 验证
  ↑                                                         ↓
  └── 用户反馈 ← 产品参谋(评估+更新知识库) ← UI(同步更新) ←─┘
```

## Gotchas

1. **产品参谋不做技术决策** — "用 Redux 还是 Zustand" 不是它的事
2. **产品参谋必须基于事实** — 读了代码才能说现状，不能凭空想象
3. **标注需用户确认的点** — 有歧义就标 `⚠️ 需用户确认`，不自作主张
4. **验收标准必须可验证** — "用户体验好" 不是验收标准，"点击按钮后 2 秒内显示结果" 才是
5. **不做什么和做什么一样重要** — Scope Out 帮团队避免范围蔓延
6. **产品参谋可以说"不需要做"** — 如果分析后发现需求不合理或已有功能覆盖，直接说
