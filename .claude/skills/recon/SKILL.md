# Recon — 竞争性侦察技能

## Overview

**闭门造车是最大的浪费。** 在写第一行代码之前，先搞清楚别人怎么做的、有什么坑、有什么可以借鉴的。

**核心升级：** 多路参谋并行侦察，不同角度交叉验证，综合决策质量远高于单参谋。

## 触发条件

以下任务**必须**先执行侦察：
- 新功能开发
- 性能/质量优化
- 架构重构或技术升级
- 涉及专业领域知识（音视频处理、AI模型、编解码等）
- 引入新框架/库/工具

**跳过侦察的例外（仅限这些）：**
- 纯 bug 修复（已有明确报错和定位）
- 纯文案/配置修改
- 用户明确说"跳过侦察"或"直接开始"

## 侦察规模（按复杂度分级）

| 复杂度 | 参谋数 | 分工 | 耗时 |
|--------|--------|------|------|
| **M 级**（2~5 文件，单域） | 1 路 | 全维度快速扫描 | 3-5 分钟 |
| **L 级**（跨域，5+ 文件） | 2 路 | 技术方案 + 风险踩坑 | 5-8 分钟 |
| **XL 级**（10+ 文件，架构变更） | 3 路 | 技术方案 + 风险踩坑 + 内部约束 | 8-12 分钟 |

### 参谋-产品（铁律第5条：功能开发必经）

**所有功能开发必须经产品参谋设计。** 覆盖范围：
- 新功能开发 → **强制**
- 改功能（逻辑/行为变更）→ **强制**
- 新 UX / 改交互 → **强制**
- 改架构逻辑（影响数据流/状态流）→ **强制**
- 改字段（数据结构变更）→ **强制**
- 设计缺陷 bug 修复（功能设计有问题）→ **强制**
- 纯代码质量修复（编译错误/typo/性能不涉及逻辑）→ 跳过

**产品知识库前置检查：**
产品参谋启动时读 `.claude/skills/product-counselor/knowledge/`，如果 `PRODUCT_BRIEF.md` 或 `FEATURE_MAP.md` 不存在或内容过时（>30天未更新），必须先执行产品接管初始化——让产品参谋从代码和文档中重建产品认知并填充知识库，然后再处理当前需求。

在任何级别的侦察中加入产品参谋。
详见 `.claude/skills/product-counselor/SKILL.md`。

```
M 级 + 产品参谋 → 1 路技术 + 1 路产品（并行）
L 级 + 产品参谋 → 2 路技术/风险 + 1 路产品（并行）
XL 级 + 产品参谋 → 3 路技术/风险/内部 + 1 路产品（并行）
```

## 参谋角色定义

### 战法库按需查询（所有参谋必读）

战法库有 40+ 条经验，全部读入浪费 token。按以下步骤按需查询：

**第 1 步：读索引**
```
Read ~/.claude/memory/tactics/INDEX.md
```
索引按 domain 分类（architecture/api/workflow/testing/debugging/collaboration），每条一句话摘要。

**第 2 步：匹配**
根据当前侦察任务的关键词，从索引中找到相关 domain 和具体条目（通常 2-5 条）。

**第 3 步：按需读详情**
只读匹配到的战法文件，不读其余的。

**第 4 步：跨项目经验查询（可选）**
如果全局战法未覆盖当前问题，查询其他项目的 observations：
```
ls ~/.claude/legion/        # 列出所有项目
cat ~/.claude/legion/directory.json  # 全局名册
# 查看其他项目的失败记录
cat ~/.claude/homunculus/observations.jsonl | grep "关键词" | tail -5
```

**铁律：禁止 `cat ~/.claude/memory/tactics/*.md` 全量读入。先索引，再按需。**

### 参谋-技术（所有级别必派）

**使命：找到最优实现方案**

**第一步：读搜索战法手册** → `Read .claude/skills/recon/references/search-playbook.md`，按领域选择对应的搜索路径。

搜索维度：
1. `"任务关键词" best practice / implementation 2026`
2. `"任务关键词" github open source stars:>100`
3. `"任务关键词" tutorial / guide`

深度侦察时（L/XL 级）：
- GitHub：star > 100 的相关项目，分析架构和可复用模块
- HuggingFace：AI/ML 相关模型和 pipeline
- 技术博客：Medium、Dev.to、知乎专栏、掘金

输出：**方案对比矩阵 + 推荐方案 + 可复用代码/模块**

### 参谋-风险（L/XL 级必派）

**使命：找到所有可能踩的坑**

搜索维度：
1. `"任务关键词" pitfalls / gotchas / common mistakes`
2. `"任务关键词" issues / bugs / known problems`
3. `"相关库名" breaking changes / migration guide`

重点关注：
- GitHub Issues 中的 open bugs 和 wontfix
- Stack Overflow 的高票踩坑回答
- 性能陷阱和兼容性问题

输出：**风险清单（按严重度排序）+ 规避方案**

### 参谋-内部（XL 级必派）

**使命：摸清当前代码库的约束和影响面**

分析维度：
1. 相关模块的调用链（谁调了它、它调了谁）
2. 相关数据结构（修改一处是否影响多处）
3. 已有的测试覆盖（哪些改动会破坏现有测试）
4. 历史决策（git log/blame 中的设计意图）

输出：**影响面分析 + 约束清单 + 建议的安全修改路径**

## 指挥官分配流程

### M 级（1 路参谋）

创建 1 个参谋 teammate（**model: "opus"**）：

```
首先执行：/effort max

你是参谋（Staff Advisor），负责技术侦察。

任务：针对「具体需求描述」执行全维度侦察。
读 .claude/skills/recon/SKILL.md 了解流程。
同时覆盖：技术方案 + 风险踩坑 + 内部约束（快速扫描）。
完成后将侦察报告通过 SendMessage 发给指挥官。
```

### L 级（2 路参谋）

**并行**创建 2 个参谋 teammate（**均 model: "opus"**）：

参谋-技术：
```
首先执行：/effort max

你是参谋-技术（Tech Scout），专注寻找最优实现方案。

任务：针对「具体需求描述」搜索技术方案。
读 .claude/skills/recon/SKILL.md 的"参谋-技术"章节。
重点：开源项目、最佳实践、可复用代码。生成方案对比矩阵。
完成后将报告通过 SendMessage 发给指挥官。
```

参谋-风险：
```
首先执行：/effort max

你是参谋-风险（Risk Scout），专注发现潜在风险和踩坑。

任务：针对「具体需求描述」搜索风险和已知问题。
读 .claude/skills/recon/SKILL.md 的"参谋-风险"章节。
重点：常见错误、兼容性问题、性能陷阱、已知 bug。
完成后将风险清单通过 SendMessage 发给指挥官。
```

### XL 级（3 路参谋）

在 L 级基础上再加 1 个（**model: "opus"**）：

参谋-内部：
```
首先执行：/effort max

你是参谋-内部（Internal Analyst），专注分析当前代码库约束。

任务：针对「具体需求描述」分析内部代码的影响面和约束。
读 .claude/skills/recon/SKILL.md 的"参谋-内部"章节。
重点：调用链、数据结构依赖、测试覆盖、历史设计决策。
完成后将影响面分析通过 SendMessage 发给指挥官。
```

## 指挥官综合决策

收到所有参谋报告后，指挥官必须：

1. **交叉验证**：参谋-技术推荐的方案是否被参谋-风险的发现否决？
2. **约束适配**：推荐方案是否与参谋-内部发现的代码约束冲突？
3. **综合报告**：合并为一份结构化决策文档

### 综合报告格式

```markdown
## 侦察综合报告

### 任务理解
（一句话概括需求）

### 侦察规模
- 级别：M/L/XL
- 参谋数：N 路
- 各参谋状态：已完成/进行中

### 技术方案（来自参谋-技术）
- 推荐方案：...
- 对比矩阵：（L/XL 级必须包含）

### 风险清单（来自参谋-风险，L/XL 级）
- 🔴 高风险：...
- 🟡 中风险：...
- 🟢 低风险：...

### 内部约束（来自参谋-内部，XL 级）
- 影响面：N 个文件/模块
- 关键约束：...

### 最终决策
- 选定方案：...
- 规避的风险：...
- 实施路径：...
```

### 侦察经验沉淀（参谋完成时必做）

侦察报告提交给指挥官后，参谋还需将关键发现沉淀为侦察战法：

通过 Bash 写入一条侦察经验到全局战法库：
```bash
cat >> ~/.claude/memory/tactics/INDEX.md << 'EOF'
- [recon-{timestamp}](recon-{timestamp}.md): {一句话摘要} — {参谋名} ({日期})
EOF
```

侦察战法文件格式（写入 ~/.claude/memory/tactics/recon-{timestamp}.md）：
```markdown
---
id: recon-{timestamp}
domain: {匹配的domain}
type: recon  # 标记为侦察经验（区别于实施战法）
score: 0
created: {日期}
source: {军团名}
project: {项目名}
summary: {一句话}
keywords: [关键词1, 关键词2, ...]
---

## 问题背景
{为什么要侦察这个}

## 调研过程
{搜索了什么、比较了什么方案}

## 关键发现
{核心洞察，别的项目最想知道的}

## 推荐方案
{最终选择的方案及理由}
```

只沉淀**有跨项目价值**的发现。纯项目内部细节不需要沉淀。

## ⚡ 狙击手自动触发检测（所有参谋必须执行）

侦察阶段是狙击手的触发器。每个参谋在检查内部知识源时，**必须**同时执行复发检测：

```bash
# 1. 在战法库中搜索与当前需求相关的关键词
grep -ril "关键词1\|关键词2\|关键词3" ~/.claude/memory/tactics/ 2>/dev/null

# 2. 在 git 历史中搜索相关修复
git log --all --oneline --grep="关键词" | head -10
git log --all --oneline --grep="fix" -- "涉及目录/" | head -10
```

**判定规则：**
- 战法库命中 + git 历史有相关 fix → **⚠️ 复发警报**，在报告中标注：`🎯 狙击手警报：此问题与战法 [tactic_名称] 相关，疑似复发`
- 仅战法库命中 → 在报告中标注：`📋 参考战法：[tactic_名称]，注意规避`
- 无命中 → 正常继续

**指挥官收到狙击手警报后：**
1. 暂停常规实现流程
2. 创建狙击手（subagent_type: "sniper"）做根因分析
3. 根因确认后再进入实现阶段

## Gotchas

1. **多路参谋必须并行创建** — 用 TeamCreate 一次创建多个，不要串行等待
2. **不要只搜一次就出报告** — 每个参谋至少搜 2-3 个不同角度
3. **不要只看 README** — 深入看 issue、PR、源码结构
4. **情报库挂了不要卡住** — curl 加超时，挂了就跳过
5. **报告要具体不要泛泛** — "用 XXX 库"没用，要说"XXX 库的 YYY 模块可以直接处理 ZZZ"
6. **战法库是最高价值源** — 团队实战经验，优先级高于外部搜索
7. **参谋-风险找到的高风险项必须在最终决策中被显式 addressed** — 不能忽略
