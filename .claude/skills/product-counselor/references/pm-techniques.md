# PM Techniques Reference（产品参谋工具箱）

产品参谋在分析需求时可按需使用以下技术。不是每个需求都用全部，根据需求复杂度选择。

---

## 1. Pre-mortem（事前验尸）

**适用场景：** 功能上线前，假设它已经失败了，倒推失败原因。

**执行方法：**
1. 假设这个功能上线 1 个月后被判定为失败
2. 列出所有可能的失败原因（至少 5 个）
3. 按概率和影响排序
4. 为 Top 3 失败原因设计预防措施

**输出格式：**
```
### Pre-mortem 分析
假设功能上线后失败，最可能的原因：
1. [原因] — 概率: 高/中/低，影响: 高/中/低 → 预防: [措施]
2. ...
```

**价值：** 比事后复盘成本低 10 倍。在产品设计阶段就规避问题。

## 2. Problem Statement（问题定义）

**适用场景：** 需求模糊时，先精确定义问题。

**模板：**
```
【谁】在【什么场景下】遇到了【什么问题】，
导致【什么后果】。
我们需要【解决方向】，使得【期望结果】。
```

**反例：** "我们需要一个导出功能" — 没有定义谁、为什么、什么场景。
**正例：** "用户在完成视频制作后，需要将所有素材和项目文件打包交付给剪辑师，目前只能手动逐个拷贝，效率低且容易漏文件。我们需要一键打包导出功能，使得用户可以直接将完整项目交付给下游。"

## 3. Success Metrics（成功标准）

**适用场景：** 定义功能的可量化成功标准，作为验收依据。

**SMART 原则：**
- **S**pecific — 具体（不是"用户体验好"，而是"操作步骤 ≤ 3 步"）
- **M**easurable — 可测量（有数字或可观测行为）
- **A**chievable — 可实现（当前技术栈能做到）
- **R**elevant — 相关（和需求目标直接相关）
- **T**ime-bound — 有时限（"加载 ≤ 2 秒"）

**输出格式：**
```
### 成功标准
- 核心指标: [描述] — 目标值: [数字]
- 体验指标: [描述] — 目标值: [数字]
- 兜底指标: [描述] — 底线: [数字]
```

---

## 4. User Journey Mapping（用户旅程）

**适用场景：** 涉及多步交互的功能，需要梳理完整操作路径。

**执行方法：**
1. 列出用户从"想做某事"到"做完"的每一步
2. 每步标注：用户的操作、系统的响应、用户的情绪
3. 找出 Pain Points（痛点）和 Delight Moments（惊喜点）

**输出格式：**
```
### 用户旅程
步骤 | 用户操作 | 系统响应 | 痛点/机会
1    | ...      | ...      | ...
2    | ...      | ...      | ...
```

---

## 5. Scope Boxing（范围框定）

**适用场景：** 防止需求蔓延，明确做什么和不做什么。

**MoSCoW 方法：**
- **Must Have** — 没有就不能发布
- **Should Have** — 重要但可以晚一个版本
- **Could Have** — 锦上添花
- **Won't Have** — 明确不做（本次）

**价值：** 团队对齐范围，避免实现者"顺手"加功能导致延期。
